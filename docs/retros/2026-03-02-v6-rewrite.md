# Retro: v6.0.0 Elite Rewrite

_Date: 2026-03-02_

## What Went Well

- **Research phase was thorough**: Competitor analysis revealed key differentiators (suggestion stripping) and confirmed market position (#1 with 2.5x downloads)
- **Clean dependency removal**: Dropped ALL runtime deps — lodash, app-root-path, AND eslint-rule-composer. Zero dependencies.
- **`export = plugin` pattern**: Eliminated the fragile sed post-processing entirely
- **Own rule wrapping is simpler and more robust**: `Object.create(context, { report })` is 15 lines vs depending on an unmaintained library. Handles both fix and suggest stripping cleanly.
- **ESM default export fix was a production bug**: unicorn v63+ rules were silently not loading due to `{ __esModule: true, default }` wrapping. Fixed in `safeRequire`.
- **Test architecture split was the right call**: Jest for fast behavioral tests, Node.js native `node:test` for ESM integration. Each plays to its strength.
- **All 28 tests pass with ESLint 10.0.2**

## What Was Surprising

- **eslint-rule-composer is broken on ESLint 10**: Uses `context.getFilename()` and `context.getSourceCode()` which were removed. Every plugin depending on it (including our competitors) is broken.
- **Jest can't handle require(esm)**: Node.js 22+ supports `require()` on ESM modules, but Jest's VM sandbox doesn't. This required the test split.
- **ESM default export wrapping**: `require()` on an ESM module with `export default` produces `{ __esModule: true, default: { rules } }`. This was causing unicorn to load 0 rules.
- **@babel/eslint-plugin doesn't support ESLint 10**: Had to replace with @stylistic/eslint-plugin for scoped plugin testing.
- **The dist/ directory had a completely different codebase**: The published dist was a feature-rich version with caching, logging, stats, and testing utilities that didn't match the source at all.

## Learnings

1. **Don't depend on unmaintained libraries for core logic**: eslint-rule-composer hasn't been updated in 7 years. Our own implementation is simpler, more correct, and handles ESLint 10.
2. **`Object.create(context, { report })` is the correct pattern**: Creates a proxy-like wrapper without touching the original context object. ESLint's rule runner doesn't care about the prototype chain.
3. **ESM unwrapping is essential for modern plugins**: Any plugin using `export default` needs the `__esModule` detection. This will only become more common.
4. **Test at the boundary, not just in unit**: The integration tests caught the ESM issue that Jest unit tests couldn't (due to VM sandbox limitations).
5. **`rootDir` in tsconfig controls output structure**: Setting `rootDir: "./src"` makes `src/index.ts` compile to `dist/index.js` (flat) instead of `dist/src/index.js`.
